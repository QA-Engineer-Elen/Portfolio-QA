## Портфолио проектов курса “Инженер по тестированию”

### Проект первого курса «Основы тестирования»
<p><strong>Проект:</strong> расписание личного кабинета для учителей Skyeng.</p>

<p><strong>Описание:</strong> в расписание учителей добавлена новая функциональность — личные события. Преподаватель может использовать личные события для собственных встреч, которые не привязаны к ученикам платформы Skyeng. Для тестирования была составлена необходимая тестовая документация в формате тест-кейсов и чек-листов. По завершении тестирования были оформлены баг-репорты, а на основе результатов создан отчет с использованием метрик и выводом о готовности продукта к релизу.</p>

<p><strong>Основная функциональность:</strong> возможность преподавателей использовать личные события для собственных встреч.</p>

<p><strong>Цель проекта:</strong> протестировать новую функциональность, повысить качество продукта, обеспечив своевременное обнаружение багов, и вынести решение о готовности продукта к релизу.</p>

**Виды тестирования, которые были проведены:**

1. **Smoke-тестирование.**<br> *Обоснования выбора:* необходимо провести для начальной оценки нового элемента - личные события, для определения общего состояния качества нового элемента. Данное тестирование позволяет проверить готовность разрабатываемого нового элемента к дальнейшему более расширенному тестированию.
2. **Функциональное тестирование.**<br> *Обоснование выбора:* выполняется чтобы убедиться, что каждая функция нового элемента системы (личные события) ведёт себя так, как указано в документации к проекту с требованиями.
3. **Регрессионное тестирование.**<br> *Обоснование выбора:* необходимо провести данное тестирования для того, чтобы проверить, что остальной функционал “Расписания” не сломался. То есть, чтобы быть уверенным, что дополнение новой функциональности (добавление элемента “личные события”) не внёс или не активизировал ошибки в областях, не подвергавшихся изменениям.
4. **Приёмочное тестирование.**<br> *Обоснование выбора:* необходимо провести для определения уровня готовности нового элемента системы / новой функции “личные события” к последующей эксплуатации.

**Какие инструменты были применены:**

- Miro для создания декомпозиции новой функциональности;
- Qase, Sitechko для написания тестовой документации;
- Confluence для ведения тест-плана и отчета;
- Jira для фиксирования баг-репортов.
